What's new in v6.4.0

  • Approval notifications for Release Pipeline participants
  • Platform Blueprint editor enhancements
    • Managed Servers are now allowed to be defined in Platform Blueprints so that repeated configuration can be avoided within the Platform Models. The reduced configuration overhead saves time and reduces the risk of making mistakes.
    • Vertical scaling is now supported for vanilla WebLogic Managed Servers. This means that on a single host and within a cluster there can be multiple Managed Servers.
    • Additional vanilla WebLogic Managed Servers and Machines can now be manually added directly within Platform Blueprints after the wizard has completed.
    • Platform Blueprints can now be created for Oracle Business Intelligence Enterprise Edition 12c with the Oracle Webtier fronting the inbound requests
    • It is now possible to target a Foreign Server to a Sub-deployment under Messaging Configuration.
    • MyST now allows for a Custom Product Catalog when defining a Platform Blueprint. This is useful when needing to extend MyST to support a target deployment environment that is not already supported out-of-the-box. When used in combination with the Deployment SDK, this feature allows Application Release Automation for any type of change be it on-prem/cloud, oracle/non-oracle, infra/app etc.
  • Platform Lifecycle Management for Axway API Gateway
    • Automated Platform Provisioning Support
    • Start, Stop, Restart & Terminate Support
  • Usability enhancements
    • When doing a Bulk Edit in the Blueprint/Model an auto-save is now performed to prevent an accidental loss of properties when a user navigating away from the component view
    • Integration Cloud Deployment Automatic Retry
    • Infrastructure Provider SSH connection tests can be run without having to first save the host details
    • Security Provider now supports Max Group Membership Search Level, Max Group Hierarchies In Cache, and Group Hierarchy Cache TTL
    • Server groups for Managed Servers can now be defined in the Platform Blueprint
    • The JNDI name is marked as mandatory for JCA Adapters
    • Connection Factory Interface is auto-calculated for each given JCA Adapter type when it is saved
    • Platform Instance will now display more than 15 pending actions
    • And more.
  • Operational improvements
    • Resolved the unwarranted display of safe warn messages in the log that were causing undue alarm
    • A new backup-database-minimal.sh script is provided to allow for easily generating a minified and redacted copy of user data in the case of raising a MyST support ticket.
  • Bug Fixes. See release notes for more information.

Installation in 3 Steps

Step 1: Run the MyST Installer

java -jar fc-installer-6.4.0.jar

Step 2: After running the installer, MyST can be started with

/opt/myst-studio/bin/start.sh

Step 3: By default, the MyST Studio console is accessible from

https://localhost/console

Behind an Internet Proxy?

If you are behind an internet proxy, please follow these steps to ensure all installation dependencies can be directly pulled from the Rubicon Red Docker Registry.

Alternatively, the image is available for direct download

MyST Studio Image 6.4.0


Upgrading to a newer release

A stable release can be auto-upgraded to a newer stable release or a newer release candidate.

Similarly, an edge release can be upgraded to a newer release candidate or even a newer stable release.

Irregardless of release type the upgrade process is always the same.

/opt/myst-studio/bin/upgrade.sh [version]

To allow for a seamless downgrade, always be sure to keep a copy of the backup file created during upgrade execution.


Reverting to a previous release

Any release (stable or edge) can be simply reverted to a previously installed version in two steps

Step 1: Retag the old version

/opt/myst-studio/bin/pull.sh [previous-version]

Step 2: Switch to the database backup that was auto-created during the prior upgrade

/opt/myst-studio/bin/update-database.sh [backup-file]
